# Artifact Contracts

This workbench treats project state as source code. Preserve these paths unless the user explicitly requests a migration.

## Work Project Root

Expected root files:

- `project.yaml`
- `AGENTS.md`
- `agentread.yaml`

Expected folders:

- `canon/`
- `characters/`
- `plot/`
- `style/`
- `scenes/`
- `drafts/`
- `reviews/`
- `memory/`
- `branches/`
- `prompts/`
- `exports/`
- `releases/`
- `tests/`

Optional generated Agent run folder:

- `agents/`

## Global Config

Default path:

```text
%USERPROFILE%\.lew\config.json
```

Override path:

```text
LEW_CONFIG_PATH
```

The global config stores active provider profile, model name, API base, default work project root, default style library root, temperature, token limit, and timeout. It may store a local profile `api_key` when the user explicitly saves it through the local frontend, but previews and API responses must redact it.

HTTP model provider entry points read this config through `model_config.py`. `LEW_MODEL_*` environment variables may override the active config for temporary experiments.

## Style Library And Style Skills

Default style library:

```text
%USERPROFILE%\.lew\style-library
```

The global config key `defaults.style_library_root` may point elsewhere.

Author-centered style projects live under:

```text
style-library/authors/{author_id}/
```

Important files:

- `author.json`
- `works/{work_id}/work.json`
- `works/{work_id}/sources/raw/*.txt`
- `works/{work_id}/sources/normalized/*.txt`
- `works/{work_id}/sources/chunks/{source_id}/*.txt`
- `profiles/{profile_id}/style-profile.md`
- `profiles/{profile_id}/style_metrics.json`
- `profiles/{profile_id}/style_prompt.md`
- `style_skills/{style_id}/style_skill.json`
- `style_skills/{style_id}/STYLE.md`
- `style_skills/{style_id}/prompt.md`

Mounted style skills inside a creative project:

```text
style/mounted/{style_id}/
style/active_style_skill.json
```

Mounted `prompt.md` has highest priority for expression-level writing choices, but it cannot override canon, character facts, plot causality, legal/safety boundaries, or explicit user constraints.

## Character Files

Character files live in `characters/{character_id}.yaml`.

Important fields:

- `character_id`
- `name`
- `role`
- `identity`
- `background_story`
- `bdi`
- `psychology`
- `relationships`
- `speech_style`
- `arc`
- `state`

`background_story` is internal behavioral causality. It should affect choices, avoidance, misreadings, tone, and relationship pressure, but should not be directly explained in final prose unless the selected scene is meant to reveal the past.

## Context Packets

Generated by `context`.

Path:

```text
memory/context_packets/{scene_id}.md
```

Purpose:

- compact working memory;
- hard canon summary;
- character state;
- style constraints;
- soft retrieval results.

Soft retrieval never overrides canon.

## Style Prompt Assets

Generated by `style-prompt` from a compiled style profile.

Files:

```text
style/{profile}/style_prompt.md
style/{profile}/style_prompt.prompt.json
```

`style_prompt.md` is the LLM-facing style constraint prompt. It is the final style-learning asset for generation, while `style_metrics.json` and `style-profile.md` are evidence and intermediate artifacts.

Generated by `style-prompt-eval`:

```text
style/{profile}/evaluation_results/{mode}/style_prompt_candidate_{timestamp}.txt
style/{profile}/evaluation_results/{mode}/style_prompt_candidate_{timestamp}.prompt.json
style/{profile}/evaluation_results/{mode}/style_eval_{timestamp}.md
style/{profile}/evaluation_results/{mode}/style_eval_{timestamp}.json
```

These outputs evaluate prompt effectiveness through provider-generated candidates.

## Local Creative Director Runs

These artifacts are produced only when using the legacy/local `director-chat` command or `/director/chat` API. In the project-type skill architecture, Codex/Claude should normally act as the Creative Director directly and may use these files only as historical project memory or local regression artifacts.

Directory:

```text
director/runs/{run_id}/
```

Files:

- `agent_decision/input.prompt.json`
- `agent_decision/raw_output.md`
- `agent_decision/parsed_output.json`
- `agent_decision/schema_validation.json`
- `tool_loop.json`
- `agent_observe_{step}/input.prompt.json`
- `agent_observe_{step}/parsed_output.json`
- `agent_observe_{step}/schema_validation.json`
- `director_decision.json`
- `director_report.md`

Index:

```text
director/runs/index.jsonl
```

The Creative Director may delegate to `project-seeding`, `character-lab`, `worldbuilding-lab`, `outline-lab`, or `scene-loop`. It records secondary decisions and risks, runs a bounded `director_tools` tool loop, observes each tool result, and then decides whether to continue or stop. It does not directly overwrite canon.

In the project-type skill architecture, these local director artifacts are not the final authority. The tool-layer agent that loaded the skill must review `director_decision.json`, `tool_loop.json`, delegated agent outputs, and reports before accepting any recommendation, asking the user, or promoting material.

## Agent Runs

Generated by `agent-run`.

Directory:

```text
agents/runs/{run_id}/
```

Files:

- `input.prompt.json`
- `raw_output.md`
- `parsed_output.json`
- `validation_report.md`
- `schema_validation.json`
- `repair_attempts/{timestamp}/`

`input.prompt.json` records provider, model label, system/user messages, metadata, and environment variable names. It must not contain API keys.

`raw_output.md` is the unmodified provider output. `parsed_output.json` is machine-readable only when parsing or later schema validation succeeds. Agent output is evidence, not canon, and must not directly write drafts, character files, releases, or `canon/`. Parsed JSON is still a draft until the platform agent reviews it against schema, canon, character, style, and user-intent constraints.

Schema files live in:

```text
schemas/agent_outputs/*.schema.json
```

Specialized agent outputs:

```text
reviews/agent/{scene_id}_scene_review.md
reviews/agent/{scene_id}_scene_review.json
reviews/agent/canon_review.md
reviews/agent/canon_review.json
agents/patch_plans/{target}_patch_plan.md
agents/patch_plans/{target}_patch_plan.json
agents/committee/{committee_id}/
reviews/agent/committee_{subject}.md
reviews/agent/committee_{subject}.json
reviews/agent/demo_walkthrough.md
```

Patch plans are candidates only. They must not be applied without a separate approval/writeback command.

## Roleplay Simulation

Generated by `simulate-scene`.

Path:

```text
branches/{scene_id}/roleplay_simulation.md
```

Contains:

- Character Agent prompts;
- World Agent consequence section;
- Branch A/B/C candidate areas;
- Director Agent scoring table;
- Canon Auditor checklist;
- writeback candidates.

It is not final prose. The platform agent must interpret the roleplay and consequence notes before selecting, revising, or discarding any branch.

## Branch Simulation

Generated by `branch-simulate`.

Directory:

```text
branches/{scene_id}/
```

Files:

- `branch_simulation.md`
- `branch_manifest.json`
- `branch_selection.md`

Branches are scored candidates, not canon. The recommended branch is a heuristic suggestion until the platform agent evaluates it and a human or approved tool-layer decision records the actual decision.

## Scene Composition

Generated by `compose-scene`.

Markdown:

```text
drafts/compositions/{scene_id}_composition.md
```

JSON:

```text
drafts/compositions/{scene_id}_composition.json
```

Contains:

- selected branch and source;
- scene beats;
- character subtext map;
- dialogue intents;
- sensory palette;
- prose seed;
- revision targets;
- writeback candidates.

It is not final prose. It is the handoff between branch reasoning and scene drafting or provider generation. The platform agent must review composition JSON before using it as a prompt pack or writeback source.

## Candidate Promotion

Generated by `promote-candidate`.

Files:

```text
drafts/promotions/{scene_id}_promotion.md
drafts/promotions/{scene_id}_promotion.json
```

Promotion records the selected candidate and creates a standard draft for review. It does not confirm canon or write character files.

## Character State Patches

Generated by `state-evolve` or by the scene-loop workflow runner after a scene draft exists.

Markdown:

```text
characters/state_patches/{scene_id}_state_patch.md
```

JSON:

```text
characters/state_patches/{scene_id}_state_patch.json
```

The patch is a pending candidate only. It records proposed known facts, state changes, arc changes, relationship changes, unresolved items, source artifact, and guardrails. It must not be treated as an automatic writeback to `characters/*.yaml`.

## Character State Apply

Generated by `state-apply` after approval.

Markdown:

```text
characters/state_patches/{scene_id}_state_apply.md
```

JSON:

```text
characters/state_patches/{scene_id}_state_apply.json
```

The apply record points to the source patch and approval run. It may update only `characters/*.yaml` state, arc, relationships, and memory refs. It must not write `canon/`.

